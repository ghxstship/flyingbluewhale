# SmartSuite Parity 01 — Fields & Formulas

Gap analysis comparing **SmartSuite's field & formula primitives** ([76 fields collection](https://help.smartsuite.com/en/collections/2614721-smartsuite-fields), [11 formulas collection](https://help.smartsuite.com/en/collections/2699469-smartsuite-formulas)) to ATLVS Technologies (`flyingbluewhale`). Repo state: 167+ Postgres tables, ~33 UI primitives in `src/components/ui/`, no generic field-type system, no formula engine.

---

## 1. Executive summary

1. **No generic field-type abstraction exists.** ATLVS encodes each domain as a hand-rolled Postgres table + bespoke server action + bespoke React form. SmartSuite's whole model — 30+ user-configurable field types living on a `table → record → field` runtime — has no analogue here. A single generic record/field system is a multi-quarter undertaking; the more realistic path is a **typed field-component library** that domain forms compose from.
2. **No formula engine.** Zero references to `formula`, no expression parser, no `IF/CASE/SUMIF`. `src/lib/compare.ts` even labels formula columns as "custom field" — i.e. acknowledged TODO. This is the single largest gap; SmartSuite ships ~80 functions ([Functions & Operators](https://help.smartsuite.com/en/articles/8686451-formula-functions-operators)).
3. **Calculated fields (Count / Rollup / Lookup) are unimplemented as primitives.** Some screens compute totals server-side via SQL aggregates, but there's no reusable "rollup field" abstraction with optional [filter conditions](https://help.smartsuite.com/en/articles/7852394-conditional-counts-lookups-rollups).
4. **Rich-text is stub-grade.** `src/components/Markdown.tsx` is a from-scratch read-only renderer (no editor, no slash commands, no embeds) vs SmartSuite's [SmartDoc](https://help.smartsuite.com/en/articles/5121698-smartdoc-field) (full Notion-tier editor: tables, callouts, code, mentions, embedded video, /commands, collaborative mode).
5. **Signature is the only "premium" field already shipped.** `src/components/SignatureField.tsx` (typed + canvas modes, dataURL hidden input) actually slightly exceeds SmartSuite's [Signature field](https://help.smartsuite.com/en/articles/4617002-signature-field) docs in spec clarity; legal metadata (timestamp, IP, geolocation) still missing on both sides.
6. **Status / Due Date / Dependency are domain-coded, not configurable.** Tickets, deliverables, RFIs, submittals each define their own status enum directly in SQL. SmartSuite's [Status field](https://help.smartsuite.com/en/articles/4612960-status-field) + [Due Date](https://help.smartsuite.com/en/articles/4612563-due-date-field) + [Dependency](https://help.smartsuite.com/en/articles/8609918-dependency-field) trio gives the user runtime control of the lifecycle — we'd benefit from at least the linked Status⇄DueDate dynamic indicator (smiley/orange/red/green-dot) on tickets and deliverables.
7. **Address field is a flat `text` column.** `clients.address`/`city`/`country` (`supabase/migrations/20260416_000007_business_tables.sql`) — no Google Places autocomplete, no `country_code`, no map preview. SmartSuite's [Address field](https://help.smartsuite.com/en/articles/4602675-address-field) breaks this into structured components from day one.
8. **Currency field has half the implementation.** `MoneyInput` (`src/components/ui/MoneyInput.tsx`) handles cents serialization beautifully, but currency code, locale-aware separators, and large-number abbreviation (1.2M) per SmartSuite's [Currency field](https://help.smartsuite.com/en/articles/4586301-currency-field) are absent.
9. **Phone field is a plain `<Input type="tel">`.** No country code + flag picker, no type tag (Office/Mobile/Fax) per [Phone field](https://help.smartsuite.com/en/articles/4602359-phone-field).
10. **Multi-language / RTL field UX is partly there.** `useFormatters` + `LocaleProvider` are wired, `start-`/`end-` (logical) Tailwind classes used in `Input.tsx` — good RTL hygiene. But there's no per-field `help_text`/tooltip primitive in the way SmartSuite ships ([Field help text](https://help.smartsuite.com/en/articles/4808448-adding-modifying-field-help-text)) — `Input.tsx` does have `hint` only, no rich-text 700-char help body.

**Recommendation:** treat formula engine and generic field-config schema as P0; pick off the 6–8 high-value field component upgrades (Address, Phone, Currency multi-currency, Rating, RichText, Checklist, Status/DueDate, Dependency) opportunistically as P1.

## 3. Formula engine

**Status: not started.** `src/lib/compare.ts` line: `"Column types map: status → status, people → members, date → due_date, formula → custom field"` — the formula type is explicitly punted to "custom field" (i.e. not mapped). No expression parser, AST, or evaluator anywhere in `src/`.

### Target: SmartSuite's function library

Per [Functions & Operators](https://help.smartsuite.com/en/articles/8686451-formula-functions-operators), ~80 functions across:

- **Operators:** `+ - * /`, `== != < <= > >=`
- **Logical:** `AND OR NOT IF IFERROR IFNONE CASE`
- **Text:** `CONCAT CONTAINS LEFT RIGHT MID LENGTH LOWER UPPER TRIM REPLACE FIND RFIND LIKE REPT ARRAYJOIN ARRAYUNIQUE T TEXT ENCODE_URL_COMPONENT`
- **Number:** `ABS ROUND ROUNDUP ROUNDDOWN CEILING FLOOR EVEN ODD MOD POWER SQRT EXP LOG NUMBER`
- **Aggregate:** `SUM AVG COUNT COUNTA COUNTALL COUNT_DISTINCT MAX MIN MEDIAN STDEV NUMERICVALUE`
- **Conditional aggregate:** `SUMIF AVGIF COUNTIF COUNTIF_DISTINCT CHECKLIST_COUNT`
- **Date:** `TODAY NOW DATE DATEADD DATEDIFF DAY MONTH YEAR QUARTER WEEK WEEKDAY HOUR MINUTE DATE_FORMAT WORKDAYS WORKDAYS_DIFF`
- **Lookup/relations:** `FIRST TOP GET_LIST RELATED_RECORD_SORT RELATED_RECORDS_SORT`
- **System:** `BLANK IS_NOT_NULL RECORD_ID LAST_MODIFIED_BY LAST_MODIFIED_TIME`

### Restrictions ([Formula Restrictions](https://help.smartsuite.com/en/articles/4757633-formula-restrictions))

- Can't reference SmartDoc / Open Comments fields.
- Linked-record dot-notation max depth = 3.
- IF branches must return same type.
- Lookup fields return arrays — must be wrapped in aggregate functions.

### What we'd need to build

1. **Tokenizer + parser** (Pratt parser is the obvious shape — operators with precedence). Tree-sitter or a hand-rolled `nearley`/`peggy` grammar.
2. **AST + type system** with three return types (`Number | Date | Text`) and a fourth runtime sentinel (`Error`).
3. **Field reference resolver** — `[Field Name]` resolves against the current row schema; `[Linked].[Field]` walks the FK graph (capped at 3, like SmartSuite).
4. **Evaluator** — pure function `(formula, row, ctx) => Value`. Stateless. Executable client-side (preview in editor) and server-side (computed columns / generated columns).
5. **Persistence:** computed Postgres view per record type, OR cache the value in a JSONB column with a recompute trigger when source columns change.
6. **Editor UX** — ~`<FormulaInput>` with autocomplete (field tokens + function signatures + inline help, like SmartSuite's Help Panel and `#` comments).
7. **Format directives** ([Formatting Formula Results](https://help.smartsuite.com/en/articles/10258278-formatting-formula-results)) — separate display config (precision, currency code, date format) from the expression itself.

**Scope estimate:** XL (8–14 weeks for one engineer) for a usable MVP covering the operators + ~30 most-used functions + linked-record depth-1.

**Quick-win subset (M, ~2 weeks):** ship a server-side-only TypeScript expression evaluator with `IF / SUMIF / DATEDIFF / TODAY / CONCAT / ROUND` and a non-editable "computed column" attribute on `xpms_atoms`. Defer the visual editor.
