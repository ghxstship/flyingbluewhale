# Surface Spec #2 — Productions (fab + install lifecycle)

**Shell:** ATLVS (primary) ↔ COMPVSS (field bridge) · **Route:** /console/production/fabrication (list) + /console/production/fabrication/[orderId]/\* (detail) + /console/punch (install-phase work granularity) + /m/punch (field)
**Status:** Drafted · awaiting review — stop signal per brief
**Theme:** Bermuda Triangle only. Accent: ATLVS pink on console surfaces, COMPVSS yellow on /m surfaces; specs reference accents by `--org-primary` token.

## 1. Data class & lifecycle

| Item                                     | Value                                                                                                                                                                                                                                                                                                                                                                                                |
| ---------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Conceptual XPMS class                    | **4 BUILD** (everything physically erected on site — site ops, scenic, construction, installation, wayfinding, tents, structures) + **5 PRODUCTION** (show systems — staging, rigging, power) for cross-class items per `docs/decisions/ADR-0004-xpms-native-nav.md` Axis A                                                                                                                          |
| Primary lifecycle (macro arc)            | **`production_phase`** — canonical LDP §2. Implemented today at `fabrication_orders.production_phase`. 9 states: `DISCOVERY → CONCEPT → ENGINEERING → PRE_PRO → FAB → LOGISTICS → INSTALL → STRIKE → ARCHIVED`. | Secondary lifecycle (workflow execution) | **`status`** — `fabrication_orders.status` (text: `open / in_progress / blocked / complete`). Cyclical within the current phase. | Install-phase work granularity           | **`punch_items`** — within `production_phase = INSTALL/STRIKE`, each gap-to-show-ready is one `punch_items` row with its own `status` (`open / in_progress / ready_for_review / complete / void`) + `show_ready_gate` boolean + `priority` (`low / normal / high / urgent`).                                                                                                                         |
| Strike-phase return granularity          | **`asset_movements`** / equipment movement events (covered in detail by Surface #3 Assets). Productions surface links out, does not author.                                                                                                                                                                                                                                                          |
| Authority docs                           | `src/lib/production-phase.ts`, `reports/LDP_LIFECYCLE_AUDIT.md` §2, `docs/decisions/ADR-0004-xpms-native-nav.md` (BUILD class cell map), `feedback_8_phase_lifecycle.md` (proposal-side fab sequence aligns 1:1 with the schema enum), `XPMS_TO_ATLVS_MAPPING.md` Layer 2 — note that the fab-shop 8-phase sequence in memory is now schema-backed by `production_phase` (audit was pre-2026-05-09). |

**Two-axis truth.** Phase = "where in the build arc is this order" (sequential, gated, owns lane assignment). Status = "what's happening to this order right now" (cyclical, owns the daily worklist signal). Together they encode the full state machine. The current ATLVS console list shows **only `status`**. The current ATLVS detail page exposes **both** but in two separate UI sections that don't speak to each other. The spec unifies them.

## 3. Primary view

**Shop Floor Board (Kanban grouped by `production_phase`, super-phase grouping default-on).** Default for any user with ≥1 fab order in flight.

The 9 lanes flatten to 3 super-phase groups; the operator can toggle to "flat 9-lane" via the View Switcher control (described below).

| Super-phase | Member phases                             | Operator question this group answers                            |
| ----------- | ----------------------------------------- | --------------------------------------------------------------- |
| **PLAN**    | DISCOVERY · CONCEPT · ENGINEERING         | What are we still designing? Drawings, approvals, scope freeze. |
| **MAKE**    | PRE_PRO · FAB · LOGISTICS                 | What's on the shop floor right now? Cuts, builds, packs.        |
| **DELIVER** | INSTALL · STRIKE · ARCHIVED (collapsible) | What's in the venue / coming out / closed out?                  |

Why this is the 80% view:

- The single business decision a shop manager makes daily is **"which orders are clearing FAB this week and need to be packed/shipped?"** — a phase board answers that directly.
- Three super-lanes mirror how shop managers actually talk ("plan / make / deliver" or "design / build / ship") and fit a 14" laptop without horizontal scroll. The flat 9-lane is for the rare deep-week-out planning session.
- Within each super-lane, the 3 sub-phases are rendered as **mini-headers** (sticky on vertical scroll), with cards beneath. Visual: a thicker divider between super-lanes, lighter divider between sub-phases. Same lift token as elsewhere.
- ARCHIVED collapses by default; click to expand. Same pattern as Linear "Done" lane.

Lane layout:

- 3 super-lane columns left-to-right: PLAN · MAKE · DELIVER. Width: equal thirds. (9-lane mode: 9 columns equal width, horizontal scroll with snap.)
- Super-lane header: name (Anton, Bermuda canon CAPITAL CASE), count of orders in super-lane, sum of `budget_cents` rollup if linked to XPMS atoms (compact `$X.XK`/`$X.XM`).
- Sub-phase header (sticky inside its super-lane): phase number + name, count, "→" button to advance the entire sub-lane (admin only, see RBAC), "+" button to file a new order in this phase.
- Card chrome:
  - Top row: order identifier (mono, `font-mono text-[10px]`, secondary text) — title (Anton or DM Sans semibold, depends on density)
  - Status pill, right-justified: `<StatusBadge status={order.status} />` (existing primitive — open/in_progress/blocked/complete)
  - Secondary row: due date relative ("3d left" / "2d over") via `<DueDateBadge>` + assignee (first initial + surname OR avatar circle)
  - Left border 3px in Bermuda ink color (per canon) — the 3px border IS the lift; no drop shadow on cards (8px shadow is Dialog-only per canon)
  - When `status='blocked'`: left border switches to `--color-warning`; small "blocker note" preview if present
  - When `status='complete'` AND not yet advanced to next phase: card carries a "Advance to Next" arrow chip (visible regardless of role; click is RBAC-gated)
- Drag-drop: forward-one-phase only (e.g. ENGINEERING → PRE_PRO; not ENGINEERING → FAB). Backward one allowed within a super-lane only (PRE_PRO → ENGINEERING OK; INSTALL → LOGISTICS OK; LOGISTICS → FAB OK). Cross-super-lane backward (e.g. INSTALL → FAB) requires the detail page with an explicit reason. Mirrors `PRODUCTION_PHASE_GRAPH` in `src/lib/production-phase.ts`, exposed without re-implementing.

**Status (the secondary axis) is NOT a lane.** Status moves are via the in-card status menu (the existing button strip on the detail page condenses to a dropdown on the card). Operators who need a status-board (e.g. "show me everything blocked, across all phases") use the **Blockers View** (see Secondary Views).

- `[ Super-phase | Flat 9 ]` toggle (segmented control, sticky in URL as `?lanes=super|flat`, default `super`).

- Project (typeahead, multi)
- Department / shop (typeahead, multi) — gated on the column landing
- Status (multi — open / in_progress / blocked / complete)
- Priority (multi — only renders if the column exists; flag for Open Questions)
- Due window (preset chips: Overdue · This Week · Next Week · Beyond 14 days)

- "This Week's Shop" — phase = PRE_PRO,FAB,LOGISTICS; due ≤ 7 days; sort by due asc.
- "Blocked" — status = blocked; group by phase.
- "Install Punch" — phase IN INSTALL,STRIKE; show-ready gate = true.
- "All" — no filters.

## 9. Mobile / narrow viewport behavior

Productions has **two mobile modes**: the read-only board snapshot for managers on the road, and the field-primary punch surface (COMPVSS) for install/strike crew.

### 9.1 ATLVS console at narrow viewport

At ≤ `--breakpoint-md` (768px):

- View switcher collapses to a chip + popover (same as Surface #1).
- Board mode: super-lane mode forced on; sub-phase rows collapse to inline section headers; horizontal scroll between super-lanes with snap.
- Timeline mode collapses to vertical week-grouped list (same pattern as Projects).
- Filter chip strip becomes a single "Filters" button → opens `<Sheet>`.
- Saved-view selector → into the same Sheet.
- Bulk actions: hidden (mobile is read + per-row action only).

### 9.2 COMPVSS field surface — install + strike

`/m/punch` is the canonical COMPVSS surface for INSTALL/STRIKE-phase work. Today it reads `tasks` filtered to `assigned_to=session.userId`. **This is a defect** — punch should read `punch_items` (the canonical install-punch table), not `tasks`. The spec corrects this:

- /m/punch reads `punch_items` filtered by RLS to the caller (assignee OR on-site for the project)
- Default view: a vertical card list grouped by project, sorted by `priority` then `due_at`
- Show-ready gates pinned at top of each project group with a yellow accent strip (COMPVSS `--org-primary`)
- One-tap card actions: "Pick up" (status=open → in_progress) · "Mark Ready" (status=in_progress → ready_for_review) · "Block" (status → blocked + note prompt)
- Realtime: `RealtimeRefresh` mounted on `punch_items` filtered by `project_id IN (...)` — when a manager adds a new punch in the console, the field surface ticks the count
- Push: new high-priority punch item → push notification to assignee (`PushKind = "punch_assigned"` — kind added to the notification_kind_catalog)

`/m/handover` (existing route, contents unprobed) is the STRIKE-phase asset-return surface — Productions surface flags but doesn't author it (asset return = Surface #3 territory).

`/m/daily-log` continues to be the freeform field-capture surface, independent of the production_phase machine. Cross-link from `/m/punch` header.

## 12. Resolutions — 2026-05-24

Future-proof resolutions to every §12 question. Spec is implementation-ready.

1. **`department_id` column?** **ADD IT.** Don't drop a useful view because the schema is missing. Add `fabrication_orders.department_id uuid REFERENCES public.departments(id) ON DELETE SET NULL` in a Phase 2 pre-req migration. Reasoning: departments are real shop-floor structure (signage, scenic, AV, electrical) — inferring them from assignee→manager→team is a UI hack that breaks when assignees move. Future flexibility: a `departments` table (org-scoped, `name`, `parent_department_id`) also future-fits `crew_members.department_id`, `punch_items.department_id`, and the per-department filter chip across every BUILD/PRODUCTION surface. The migration creates the table empty; backfill is human-driven by the shop manager. Workbench view ships only when ≥1 department exists in the org (graceful degrade).
2. **`priority` column?** **ADD IT.** `fabrication_orders.priority text NOT NULL DEFAULT 'normal' CHECK (priority IN ('low','normal','high','urgent'))`. Mirrors `punch_items.priority` and `tasks.priority` (P1/P2/P3 number mapping handled at the UI layer). Reasoning: cross-table priority consistency is the unlock for future cross-surface filtering ("show me everything urgent across fab + punch + tasks"). One enum vocabulary, three call sites.
3. **XPMS atom → punch_item enumeration?** **Strongest join with graceful fallback.** Trigger logic on `fabrication_orders` UPDATE where `production_phase` transitions to `INSTALL`:
   - Resolve linked atoms: `fabrication_orders.xpms_atom_id` (if set) → recursive descent via `xpms_atoms.parent_id IS NOT NULL` → collect LEAF atoms (no children).
   - IF leaves found → INSERT one `punch_items` row per leaf, `title='Install: {atom.identifier} {atom.name}'`, `show_ready_gate=true`, `assignee_id` from the atom if present.
   - ELSE (no atom linkage or no leaves) → INSERT one stub `punch_items` row per order, `title='Install: {order.title}'`, `show_ready_gate=true`, `assignee_id=order.assignee_id`.
   - In BOTH cases: write `notification` to the assignee(s) with `kind='punch_assigned'`.
   - Trigger is idempotent — uses `ON CONFLICT (fabrication_order_id, atom_id) DO NOTHING` on a new unique index. Re-advancing into INSTALL after a regression doesn't duplicate.
   - Reasoning: the bridge works regardless of how strong atom linkage is today, and gets richer as linkage strengthens — no rewrite when the data fills in.
4. **`<StatusBadge>` rename?** **NOT THIS SCOPE.** Surface specs keep the primitive name. A separate post-Pass-1 cleanup PR will codemod 80+ call sites to `<StatePill>` / `<PhaseChip>` (split by axis, not just rename). Spec author writes `<StatusBadge>` in code references for compile compatibility but uses "state pill" / "phase chip" / "lifecycle indicator" in prose so the conceptual vocabulary leads. JSDoc note added to `StatusBadge.tsx` flagging rename intent so future authors don't reach for new instances.
5. **/m/punch table swap?** **SAFE.** Grep confirms only `src/lib/nav.ts` (label only) and `src/app/(mobile)/m/page.tsx` (router tile, label only) reference `/m/punch`. No caller depends on `tasks`-table behavior. Swap proceeds as spec'd; mobile tile copy updates from "Open items" to "Show-ready gaps" in the same PR.
6. **Manager role split?** **Reserve the namespace; don't split now.** Introduce `src/lib/auth/policy.ts` with predicates: `canAdvanceProductionPhase(session, orderRow)`, `canAdvanceProjectPhase(session, projectRow)`, `canRegressPhase(session, recordRow)`. Today each resolves to `hasOrgRole(session, ['owner','admin','manager'])`. When the role taxonomy splits (shop_manager vs project_manager vs program_manager), only the predicate body changes. Surface code never references role names directly. Same pattern applies across Surface #1 (Projects) and Surface #4 (Deliverables) — adopted as a Pass-1 convention.
7. **Reason-required-on-INSTALL-advance policy hardcoded?** **NO — make it data-driven.** New table `production_phase_advance_policy (id uuid PK, org_id uuid FK, from_phase production_phase, to_phase production_phase, reason_required boolean DEFAULT false, reason_label text, reason_hint text, UNIQUE(org_id, from_phase, to_phase))`. Seed: one row per `(org_id, LOGISTICS, INSTALL)` with `reason_required=true`, `reason_label='Ready Note'`, `reason_hint='What's done? What still needs to happen on-site?'`. SDK at `src/lib/production-phase.ts` reads the policy and surfaces it to the advance dialog. Adding a new reason-required gate tomorrow is one INSERT, zero code changes. Identical pattern is planned for `project_phase_advance_policy` (Surface #1) — both surfaces share the same row schema with a `phase_kind enum('project','production')` discriminator if we want one table; otherwise two tables. **Default: one table, discriminated** — cuts lifecycle-policy infra in half across the platform.
