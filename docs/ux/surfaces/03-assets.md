**Shell:** ATLVS (primary) + COMPVSS (scan + field-move) · **Route:** /console/production/equipment (list) + /console/production/equipment/[equipmentId]/\* (detail) + /console/production/rentals/availability (matrix) + /console/production/warehouse/inventory (rollup) + /m/wms + /m/inventory/scan (field)
**Status:** Drafted · awaiting review — stop signal per brief
**Theme:** Bermuda Triangle only. ATLVS pink on console; COMPVSS yellow on /m surfaces. Accents reference `--org-primary` token.

## 1. Data class & lifecycle

| Item                          | Value                                                                                                                                                                                                                                                                                                                                                                                                 |
| ----------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Conceptual XPMS class         | **5 PRODUCTION** (show systems: audio, lighting, video, staging, rigging, power, SFX) + **6 OPERATIONS** for logistics-leaning categories (transport, freight). Class membership lives at the asset row via `equipment.category` and (future) `equipment.xpms_class_code`.                                                                                                                            |
| Primary lifecycle (canonical) | **`ual_state`** — LDP §3 canonical name, shipped as enum at `0019_usnp_canon_local_parity.sql:40`. 9 states: `acquired → available → reserved → in_use → in_transit → returned → in_maintenance → retired \| lost`.                                                                                                                                                                                   |
| Lifecycle column today        | **`equipment.status`** (`equipment_status` enum: `available / reserved / in_use / maintenance / retired`). 5 states, sparse. LDP audit §3 flags this as PARTIAL (wrong name, missing 4 states, no ledger).                                                                                                                                                                                            |
| Movement ledger               | **`asset_movements`** — append-only, typed `from_state` / `to_state` (both `ual_state`), `from_place_id` / `to_place_id`, `from_custodian_id` / `to_custodian_id`, optional `reservation_*`, optional `reference_kind` / `reference_id` (to a fab order, project, rental, etc). Already exists per `0019:97`. Today UI does not surface it.                                                           |
| Adjacent tables               | `rentals` (booking window + rate), `maintenance_jobs` / `maintenance_schedules` (service cadence), `asset_links` (parent/child kit relationships), `locations` (depot / venue / bin geography).                                                                                                                                                                                                       |
| Authority docs                | `reports/LDP_LIFECYCLE_AUDIT.md` §3 (stale — pre-`ual_state` enum landing), `supabase/migrations/0019_usnp_canon_local_parity.sql` (canonical enum + table), `docs/XPMS_TO_ATLVS_MAPPING.md` Layer 1 UAL row, `src/app/(platform)/console/production/equipment/[equipmentId]/page.tsx` (current 5-state UI), `src/app/(platform)/console/production/rentals/availability/page.tsx` (existing matrix). |

**One-axis truth, schema fragmentation.** Unlike Projects (phase + state two-axis) and Productions (phase + status two-axis), an Asset has exactly one lifecycle axis: where it is in custody. The fragmentation isn't conceptual, it's schema-historic — `equipment.status` shipped first; `ual_state` + `asset_movements` shipped later; the UI never moved over. **This spec drives the migration of read+write paths from `equipment.status` (5 states) to `equipment.ual_state` (9 states) backed by `asset_movements`.**

## 3. Primary view

**State-filtered table with summary tile strip (Asset Panda pattern).** Default for everyone with access to /console/production/equipment.

Why a table, not a board (the divergence from Surfaces #1 and #2):

- **Volume.** Asset inventories run 500–5000 units. A 5000-card kanban is unusable; a 5000-row virtualized table is what `DataTableInteractive` was built for (`@tanstack/react-virtual` is in deps and already powers DataTable).
- **The business decision** is **"find this unit"** or **"how many of {category} are available next Tuesday?"** — both are predicate-shaped (filter + count + sort). A board over-rotates to "see the workflow."
- A board IS appropriate for a small batch — a derived view "today's incoming returns" (rows where `ual_state IN ('in_transit','returned')` AND last movement < 24h) can render as a board because that batch is bounded. Spec ships it as a `DataViewKind = 'board'` for the filtered subset only — auto-disabled when row count > 60.

Layout:

- **Tile strip** above the table — one tile per `ual_state` value showing live count. Clicking a tile toggles the filter chip for that state. Active tile gets `--org-primary` underline. Mirrors `/console/production/warehouse/inventory` rollup section but elevated to filter primary.
- **Filter chip strip** below the tiles: state (multi), category (typeahead, multi), location (typeahead, multi), assignee/custodian (typeahead), rate range, "Coming home in N days" preset chips, "Overdue back" toggle, condition (good / needs_service / out_of_service — see Open Questions on the `condition` column).
- **Table** — virtualized, sticky-header. Default columns: name · category · asset_tag (mono) · serial (mono, default hidden) · ual_state pill · current location · current custodian · daily rate · last movement (relative) · upcoming bookings (count). Operator can reveal/hide via DataTable column menu (already supports per-column hidden).
- Sort: name asc default; click-to-sort everything; multi-sort via Shift-click (already supported).
- Row click → equipment detail.
- Row hover → row actions menu reveals: Scan to move (opens scan-or-pick dialog) · Reserve · Send to maintenance · Check out · Print QR.

## 9. Mobile / narrow viewport behavior

**Scan flow is mobile-primary.** Field crew never sees the ATLVS console for this surface — they live in `/m/wms` + `/m/inventory/scan`.

### 9.1 ATLVS console at narrow viewport

At ≤ `--breakpoint-md` (768px):

- Tile strip wraps to 3 tiles per row.
- Filter chips → "Filters" sheet.
- Table: hide `category` and `rate` columns; pin `name` + `ual_state`. Use compact density default.
- Matrix: degrades to 3-day window (today, +1, +2) with horizontal scroll; row count stays 50 max.
- Map: hidden, replaced with "Use a larger screen for the map view." Same rule as Surface #1.
- Bulk operations: hidden.

### 9.2 COMPVSS field surface

`/m/wms` (existing) becomes the field landing. Today it shows three sections: scan · maintenance flagged · rentals due back. Spec extends:

- Top: large primary "Scan" button (existing) — opens `/m/inventory/scan`.
- Section 1: "Coming home today" — units whose latest movement has `to_state IN (in_transit, returned)` OR `rentals.ends_at <= today + 1`. Tap → detail with quick "Mark Returned" action.
- Section 2: "On my person" — units checked out to `session.userId` (custodian filter). Tap → quick "Hand off" or "Mark Returned".
- Section 3: "Service flagged" — `ual_state = in_maintenance`. Tap → maintenance jobs list.
- Realtime: `RealtimeRefresh` mounted on `asset_movements` filtered by org → live updates as other field crew scan.
- Push: assigned unit reaches `to_state=in_transit` heading to my venue → push notification.

`/m/inventory/scan` (existing, has `InventoryScanner.tsx`) — extend the post-scan flow:

1. Scan → resolve to unit. If not found → "Unknown tag" toast.
2. Unit resolved → bottom sheet with current state + allowed transitions (filtered by `canMoveAsset()`).
3. Operator selects transition.
4. Optional: select destination location (preset list from `locations`, pre-sorted by proximity if geolocation permitted) or destination custodian (pre-filtered to current project's crew).
5. Optional: scan reference doc QR (fab order, rental, project) to attach `reference_kind` / `reference_id`.
6. Confirm → write `asset_movements` row → toast "{unit.asset_tag} → {toState}".
7. Stay on scanner screen for the next scan (bulk loop).

Bulk-scan loop is the same flow but with the target state chosen once at the top of the loop ("Mark next 12 scans as RETURNED to {location}"). Each scan auto-applies and goes to the next. Bottom counter: "12 scanned · last: {asset_tag}".

## 12. Resolutions — 2026-05-24

Future-proof resolutions baked in from the start of this spec. Where a downstream question surfaces in implementation it is captured here, not deferred.

1. **`equipment.ual_state` column?** **ADD IT.** Cleanest path is a single migration that (a) adds the column, (b) backfills from `equipment.status` 1:1, (c) installs a trigger that keeps it in sync with the latest `asset_movements.to_state`. The legacy `equipment.status` column stays for one release as a fallback; a follow-up sweep PR drops it. Reasoning: writing the UI against the canonical column today avoids a second rewrite when the legacy column drops. The trigger means existing flows that still write `equipment.status` continue to function until the sweep.
2. **`condition` axis (good / needs_service / out_of_service)?** **DEFER, not in scope.** No schema column exists today. The maintenance state (`in_maintenance`) covers the "out of service" case. Condition-as-second-axis is a known gap (e.g. a unit can be `in_use` AND `needs_service` because the operator flagged it but hasn't pulled it from the show yet). Flagged for a future surface spec; the chip is removed from the filter strip until the column lands.
3. **Kit / sub-rental hierarchy?** **PARTIAL.** `asset_links` (parent/child) exists; sub-rental from a third-party vendor doesn't have a clean schema yet (no `sub_rental_vendor_id` on `rentals`). Tree view ships against `asset_links` for owned kits. Sub-rental is flagged in surface backlog, not specced here.
4. **Per-unit serial vs. per-SKU pool?** **Per-unit only.** Every `equipment` row is a single physical unit. Pool/quantity-based assets (e.g. 200 identical cables) would need a separate `equipment_pools` table — out of scope. Spec assumes the per-unit model and recommends 1 row per cable for now (acceptable at the scale we operate; revisit at 10k+ rows).
5. **Map view license / map tiles?** Uses `maplibre-gl` (already in deps). No additional license needed.
6. **`acquired` state — when does a new unit get one?** When the operator creates the row via `+ Add Equipment` OR via Bulk-Receive. First `asset_movements` row written has `from_state=NULL, to_state=acquired`. Backfilled equipment lacks this row; spec accepts the gap (no synthetic acquired rows for historic units).
7. **`equipment.status` rename to `equipment.ual_state` — does the UI need a deprecation banner?** No — the trigger keeps both columns coherent during the transition; the UI reads ual_state from day one. Legacy column drop is a separate PR.
8. **Custodian model — user vs crew_member?** `asset_movements.{from,to}_custodian_id` is generic uuid today (no FK constraint visible in the slice we read). Spec assumes it points at `users(id)` for individual custody and `vendors(id)` for sub-rental return. Add a `custodian_kind` discriminator column if both are needed in production. Default for the spec: `users(id)` only.
