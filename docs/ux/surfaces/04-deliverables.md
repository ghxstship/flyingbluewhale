# Surface Spec #4 — Deliverables (advancing)

**Shell:** ATLVS (admin) + GVTEWAY (counterpart) + COMPVSS (assignee) — three-shell parity over one table
**Route:** /console/projects/[projectId]/advancing (admin list) + /console/projects/[projectId]/advancing/assignments/[deliverableId] (admin detail) + /p/[slug]/<persona>/advancing (per-persona portal) + /m/advances (cross-project mobile)
**Status:** Drafted · awaiting review — stop signal per brief
**Theme:** Bermuda Triangle only. ATLVS pink on console, GVTEWAY cyan on portal, COMPVSS yellow on mobile. Accents via `--org-primary`.

**Two flavors, one machine.** Project-documents flow `briefed → draft → submitted → in_review → approved → delivered` with strong review chrome (file preview, comments, approve/reject). Catalog assignments flow the same states but their `submitted`/`approved` transitions are administrative (the operator marks the credential "approved" once the print job is back) and the meaningful event is `fulfilled_at`. The surface keeps the lifecycle unified at the column level and **diverges the UI at the kind level**.

## 3. Primary view (per shell)

The surface diverges across shells. **One spec, three primary views, all backed by the same `deliverables` rows.**

### 3.1 ATLVS admin (`/console/projects/[projectId]/advancing`)

**Primary view: Review Queue Board (Kanban grouped by `deliverable_state`).** Two top-level filter chips toggle data scope: `[ Project Documents | Catalog Assignments ]` (segmented control in the header, sticky in URL as `?scope=docs|catalog`, default = `docs`).

Why a board:

- The admin's daily question is **"what's stuck in review and needs me to unblock it?"** A board exposes stuck items by lane density.
- Aligns with Surfaces #1, #2, #3 vocabulary — the operator learns one chrome and reuses it across lifecycles.
- 8 lanes is at the upper bound of board-readability. Like Productions, we collapse to **3 super-lanes** when the operator wants the macro view: `[ INTAKE (briefed, draft) | REVIEW (submitted, in_review, revision_requested) | DONE (approved, delivered, rejected) ]`. Same `?lanes=super|flat` toggle as Surface #2.

Lane layout:

- 8 lanes left-to-right when flat: briefed · draft · submitted · in_review · revision_requested · approved · delivered · rejected
  - `revision_requested` and `rejected` are rendered with `--color-warning` / `--color-error` lane tint so they pop as needing attention
- Super-lane mode: 3 columns INTAKE / REVIEW / DONE; sub-lane headers as in Surface #2
- Lane header: count + sum-of-overdue-count (small red badge if > 0)
- Card chrome:
  - Top row: artifact-kind icon (file vs assignment-kind icon) · title · version (mono `vN`)
  - Middle: who-owns-this-state (e.g. "Awaiting your review" / "With sound team" / "Assigned to {name}")
  - Bottom row: `<StatusBadge state={d.deliverable_state} />` (primitive name unchanged, "state" prop alias added — see §10) + `<DueDateBadge>` + (for catalog kinds) `Fulfilled` chip when `data.fulfilled_at` set
  - 3px ink left border per Bermuda canon; orange/red tint for revision_requested/rejected; cream otherwise
  - On hover: row-actions reveal — Open Review · Comment · Override State · Reassign · Archive
- Drag-drop: same adjacency policy family as Surfaces #1/#2. Forward = adjacent only; backward = within super-lane only; cross-super-lane backward requires reason via detail page.
  - Exception: `submitted → in_review` is forward-and-automatic — opening the review surface flips this state ("on view = in review" Frame.io pattern). Drag still works; auto-flip handles the case where the reviewer opens the artifact without dragging first.
- **Comment thread + file preview** open in a right-rail drawer when a card is clicked (Frame.io pattern). Detail page (`/advancing/assignments/[deliverableId]`) is the dedicated full-screen surface for deeper review; the drawer is the in-context inspector.

Secondary admin view: see §4.

### 3.2 GVTEWAY portal (`/p/[slug]/<persona>/advancing`)

**Primary view: Per-kind Inbox.** The portal user is a counterpart (artist, vendor, sponsor, crew). They submit work; they don't manage the queue. Their question is **"what's mine, what's the status, what do I still owe?"**

- Header chip strip: scope filter `[ All | Open | Submitted | Approved ]` with counts.
- Vertical list **grouped by kind** (Catering, Credentials, Riders, etc — per `src/lib/db/advancing.ts` type lists) with a count per section.
- Each card: title · version · state pill · due date · last activity (relative). Tap card → per-deliverable submission surface with file upload + comment thread.
- Submission surface inherits the same right-rail comment thread as ATLVS — the comment store is the same `deliverable_comments` rows. The portal reviewer's comments appear at the admin in real time and vice versa.
- "Submit" / "Resubmit" buttons are state-aware. From `briefed` → "Begin Draft". From `draft` → "Submit". From `revision_requested` → "Submit Revision".

`<PortalDocVault>` (existing shared primitive) continues to be mounted on the persona index pages as a read-only "documents tied to me" widget. Spec doesn't change it; it complements the new submission inbox.

### 3.3 COMPVSS field (`/m/advances`)

**Primary view: Today/Soon/All Inbox (mobile)** — same shape as portal, but cross-project and tap-friendly.

- Top filter chip strip: `[ Today | This Week | All Open | Done ]`. Default `Today` (deadline ≤ today + 1 OR last-activity ≤ 24h).
- Vertical card list grouped by kind (existing pattern, kept). Each card: title · project (small mono) · state pill · due. Tap → quick-action sheet:
  - Catalog kinds: "Acknowledge" (state → approved) · "Fulfilled" (writes `data.fulfilled_at = now()`) · "Snooze 24h" (UI-only).
  - Rider kinds: "Open in browser" (deep link to portal submission surface — mobile composer is out of scope; field crew uses portal for richer work).
- Realtime: `RealtimeRefresh` on `deliverables` filtered by `assignee_id=session.userId` → live state changes.
- Push: kind = `"advancing_assigned"` (new row landing on me) + `"advancing_revision_requested"` (state flipped requiring my attention). Already plumbed in CLAUDE.md "Push fan-out" section.

Filter chips below the switcher:

- Scope (segmented): `Project Documents | Catalog Assignments | All`
- State (multi)
- Kind (typeahead, multi) — list derived from current scope
- Assignee (typeahead) — only meaningful when scope = Catalog Assignments
- Reviewer (typeahead) — only meaningful when scope = Project Documents (requires reviewer column)
- Deadline window (preset chips: Overdue · Today · This Week · Next Week · Beyond 14 days)

- "Stuck in Review" — state IN (submitted, in_review); updated_at older than 48h.
- "Revision Requested" — state = revision_requested; sort by deadline.
- "Approved, Awaiting Delivery" — state = approved AND `data.fulfilled_at IS NULL`; scope = Catalog Assignments.
- "All My Reviews" — assigned to me, state IN (submitted, in_review).
- "All" — no filters.

## 9. Mobile / narrow viewport behavior

Already covered in §3.3 (COMPVSS) and §3.2 (portal). Additional notes:

### 9.1 ATLVS console at narrow viewport

- Board → super-lane mode forced; horizontal snap-scroll between super-lanes.
- View switcher collapses to chip + popover.
- Filter chips → "Filters" sheet.

## 11. Acceptance — the business decision the surface exists for

Phase 2 lands tests asserting:

1. **Operator can advance a deliverable from `in_review` to `approved` in ≤2 clicks from any console route** (brief-mandated example).
   - ⌘K → "approve {deliverable title}" → Enter ⇒ row advances; history row written.
   - From /advancing board, single click card → drawer opens with file preview + thread → click Approve ⇒ row advances.

2. **Comments work across all three shells in real time.** Admin posts a comment on /console/projects/[id]/advancing/assignments/[deliverableId]; within 30s the artist's /p/[slug]/artist/advancing/[deliverableId] shows the comment (via `RealtimeRefresh` mounted on `deliverable_comments` filtered by deliverable_id). Same true for portal → admin direction.

3. **Auto-open-as-in-review fires.** Opening the review surface for a `submitted` deliverable flips to `in_review` and writes a `deliverable_history` row recording "auto-on-open by {user}".

4. **Catalog assignment can be acknowledged in 1 tap on mobile.** From /m/advances → tap a credential assignment card → tap "Fulfilled" ⇒ `data.fulfilled_at` set; chip appears; admin sees the chip on next paint.

5. **Bulk approve respects per-org policy.** When `deliverable_approval_policy` is set to "admin-only" for the org, a manager selecting 5 in-review deliverables and clicking "Bulk Approve" results in 0 approvals + a toast explaining the policy + notification fan-out to admin.

6. **Anchored comments survive a version bump.** Author a comment with `anchor = {page: 3, x, y, w, h}` on v1; submit v2 of the deliverable; comment thread on v2 surfaces the v1 comment as "Resolved against v1, was on page 3 area" (snapshot reference retained).

7. **Rejected reason is non-skippable.** Attempt to reject without filling the comment body ⇒ dialog inline error; no DB write.

## 12. Resolutions — 2026-05-24

Future-proof resolutions baked in. Where a downstream question surfaces in implementation it is captured here, not deferred.

1. **Project-documents vs catalog-assignments split — same surface or split surfaces?** **Same surface, scope segmented control at the top.** Reasoning: they share the lifecycle column, the comment thread, the history log, the storage bucket, and 90% of the chrome. Splitting at the URL level (`/advancing/documents` vs `/advancing/assignments`) would force duplicate save-view + filter + RBAC infrastructure. The segmented control is the conventional pattern for "two related lenses on one dataset" (Linear's Issues/Projects/Views toggle). Each lens defaults to a different view (`docs → board`, `catalog → table`) so operators land in the right shape for the right scope.
2. **Fulfillment as JSON flag vs first-class state?** **Keep as JSON flag for now; add `delivered` to the canonical enum and merge them later.** Reasoning: `data.fulfilled_at` is a JSON scalar set by ops in the field; promoting it to a state today would require a backfill and a state-machine rewrite. The compromise: surface `Fulfilled` as a first-class chip alongside the state pill (already in-place); plan a future cleanup PR that lifts `fulfilled_at` into a `delivered` state value once the JSON usage is well-understood. The 8-state enum gains `delivered` and `revision_requested` and `briefed` (per LDP audit recommendation); fulfillment_at becomes a derived `delivered_at` column at that point.
3. **Comment anchors — schema shape?** **`anchor jsonb NULL`** on `deliverable_comments`. Shape: `{type: 'pdf_region', page, x, y, w, h}` or `{type: 'image_region', x, y, w, h}` or `{type: 'video_frame', frame_ms}`. Renderer dispatches on `anchor.type`. Forward-compatible: future anchor types (timeline, audio waveform, 3D scene) are an additive shape rather than a column add.
4. **Comment versioning across deliverable versions?** **Anchor carries `version_at_post` integer.** When a new version is submitted, prior comments aren't deleted — they're rendered as "From v{N} review" with a small badge. Operator can choose to roll them forward (manually copy the comment into the new version) or resolve them.
5. **Per-org approval policy table?** **YES — `deliverable_approval_policy`.** Mirrors `phase_advance_policy` shape from Surface #2 Resolution #7. Per-org rules controlling which role can transition which state for which kind. Reasoning: hardcoding "manager can approve" leaks an opinion that doesn't match every customer (legal/contracts may want owner-only approval). Schema-driven policy is a one-INSERT extension.
6. **Auto-flip submitted→in_review on open — should it fire only for reviewers, not for casual lookers?** **Yes — gate on `canTransitionDeliverable(session, row, 'submitted', 'in_review')`.** Non-reviewers opening the surface only render; reviewers' opens flip the state. Reasoning: stops state-noise from non-actors but keeps the Frame.io frictionless review pattern for reviewers.
7. **Comment thread primitive — deliverables-only or generalized?** **Generalize.** `<CommentThread targetTable={…} targetId={…} />` so the same primitive serves future commentable records (deliverables now; punch_items, fab orders, projects later). Backing table is `deliverable_comments` for now; future-generalized `comments` table (or polymorphic `record_comments`) can absorb it via a view. Decision deferred to whichever record adds comments next — for now the primitive accepts a `targetTable` and routes to the right query.
8. **Realtime cost — `RealtimeRefresh` on deliverables + comments + history is three subscriptions per page?** **One filtered subscription per page; client merges.** Mount a single `RealtimeRefresh` filtered to `deliverable_id IN (…visibleIds)` against a new `v_deliverable_activity` view that UNIONs comments + history + state changes. Cuts subscription count 3× per page. Future-flexible: when comments-on-other-tables ship, the union view grows another arm.
9. **Reviewer column — should `deliverables` carry an explicit `reviewer_id`, or is "any admin/manager" the reviewer pool?** **Add `reviewer_id uuid NULL` (FK users).** Reasoning: an "any admin can review" model collapses when 12 active deliverables sit in `submitted` and nobody owns clearing them. Explicit reviewer assignment lets the inbox view show "your queue" and lets push notifications route precisely. Default behavior when `reviewer_id IS NULL`: state machine treats the deliverable as unassigned; admin board shows an "Unassigned reviewer" badge.
10. **Three-shell consistency — should the board view be available in the portal/mobile?** **No.** Portal + mobile are recipient-side; their question is "what's mine"; an inbox answers that better than a board. Board stays admin-only. Future flexibility: if/when org-side counterparts (e.g. a vendor with their OWN admins) need a board view of their own submissions in the portal, build it then; do not pre-build.
