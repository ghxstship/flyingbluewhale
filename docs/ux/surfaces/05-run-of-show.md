# Surface Spec #5 — Run of Show (gated checkpoint board)

**Shell:** COMPVSS (primary — calling at showtime) + ATLVS (authoring + control room) + GVTEWAY (talent set-time + standby push)
**Route:** /console/production/ros (authoring + control room) + /m/ros + /m/ros/[showId] (calling) + /p/[slug]/<persona>/cues (talent set-time view)
**Status:** Drafted · awaiting review — stop signal per brief
**Theme:** Bermuda Triangle only. COMPVSS yellow on /m surfaces, ATLVS pink on console, GVTEWAY cyan on portal. Accents via `--org-primary`. Live cue accent is a fixed `--color-error` (red) regardless of shell — universal "this is going out right now" signal.

## 1. Data class & lifecycle

| Item                  | Value                                                                                                                                                                                                                                                                                                                                                                                                                       |
| --------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Conceptual XPMS class | **5 PRODUCTION** (show systems — staging, lighting, audio, video, rigging) per `docs/decisions/ADR-0004-xpms-native-nav.md`. ROS is the canonical orchestration surface for everything in the technical envelope during show.                                                                                                                                                                                               |
| Primary lifecycle     | **`cue_state`** (LDP-canonical name) — implemented as `cues.status text CHECK (status IN ('pending','standby','live','done','skipped'))`. 5 states. Cyclical — `done`/`skipped` are terminal; `pending` ↔ `standby` ↔ `live` ↔ `done` is the run loop. LDP-banned `status` name retained per Resolution #4 in Surface #2 (rename in the codemod sweep PR, not in scope). Sample copy already calls it "status" in the page. |
| Lane axis             | `cues.lane text CHECK (lane IN ('show','lights','audio','video','talent','safety','transport'))` — 7 lanes. Drives the strip-board swimlane axis.                                                                                                                                                                                                                                                                           |
| Time axis             | `cues.scheduled_at timestamptz NOT NULL` + `cues.duration_seconds int NULL`. Pacing is derived; nothing else in the schema enforces sequencing — this spec calls out a `predecessor_cue_id` add (see §12).                                                                                                                                                                                                                  |
| Show scope            | `cues.event_id` (FK to `events`) — NULLABLE today. When set, the cue is scoped to a single event/set within a project (e.g. headliner set vs opener vs DJ change). When NULL, cue is org/project-wide (e.g. doors open). Multi-show days need this column reliably populated; spec recommends making it required at the surface level (not the schema level — too late for a NOT NULL backfill).                            |
| Ownership             | `cues.owner_id` (FK users) — NULLABLE. The department lead responsible for confirming standby (lights LD, audio FOH, etc.).                                                                                                                                                                                                                                                                                                 |
| Transition log        | **MISSING.** No `cue_history` or `cue_transitions` table. Today every status change overwrites the column. **Material gap for post-show retrospectives** ("when did we actually fire cue 47?"). Spec recommends `cue_transitions` (append-only, `from_state`, `to_state`, `fired_at`, `fired_by`) and uses it for the running cue log in show mode.                                                                         |
| Gating model          | **Not in schema.** Every cue today is implicitly gated (manual GO required). Auto-follow chains (the Q-Lab pattern — cue A automatically fires cue B after N seconds) are not modeled. Spec defines `autofollow` (boolean) + `autofollow_after_seconds` (int) + `predecessor_cue_id` (FK self) as additive columns; defaults preserve current "everything gated" behavior.                                                  |
| Authority docs        | `src/app/(platform)/console/production/ros/page.tsx`, `src/app/(platform)/console/production/ros/CueForm.tsx`, `src/app/(mobile)/m/ros/page.tsx`, `supabase/migrations/0001_remote_snapshot.sql:3068` (cues schema), `docs/decisions/ADR-0004-xpms-native-nav.md` (PRODUCTION class).                                                                                                                                       |

**Two distinct surface modes, one schema.** A cue exists across two operator contexts:

- **Plan mode** (pre-show, days/hours before): author + sequence + assign cues. The day's whole show is laid out, drag-rearranged, exported.
- **Show mode** (live, during the show): one cue is current, one is on standby, the next 5 are queued. A stage manager calls GO; cues advance; clock runs.

Same rows, same lifecycle column, **completely different chrome**. This spec defines both as primary views of the same surface, switched by a top-level mode toggle. Most other lifecycle surfaces in this pass have one chrome that adapts; ROS genuinely needs two.

## 3. Primary view (per mode)

Mode toggle is a top-level segmented control in the page header: `[ Plan | Show ]`. Persists in URL (`?mode=plan|show`). Default at /console/production/ros = `plan`; default at /m/ros = `show`.

### 3.1 Plan mode (ATLVS primary)

**Primary view: Strip Board (Studio Binder pattern).** Swimlanes = lanes; horizontal axis = time. Strips = cues. Drag to reorder within a lane; drag across lanes to reassign lane.

Layout:

- **7 swimlanes** vertically — show / lights / audio / video / talent / safety / transport (in that order, mirroring `cues_lane_check`). Lane header (sticky-left): name + count + departmental lead avatar (from `cues.owner_id`).
  - Lane header has a "Solo" button that filters the board to a single lane for department-lead authoring.
- **Horizontal time axis**: tick marks at every 15 minutes; major ticks at hourly. Day-break visual divider (thick line) at midnight; multi-day shows render multiple days on a long horizontal scroll with snap-to-day.
- **Strip card**:
  - Fixed height per lane row. Width = `duration_seconds` (default 5 minutes when null — see §12 for a `min_visible_duration_seconds` setting).
  - Top-left: scheduled time (mono, large enough to read at 100% zoom).
  - Body: cue label (one line, ellipsis on overflow). Hover → tooltip with full description.
  - Bottom: `<StatusBadge>` (state pill) + owner avatar + auto-follow chevron if `autofollow=true` pointing to predecessor cue.
  - 3px ink left border per Bermuda canon. Live cues get `--color-error` border on top of the ink for highest-contrast attention.
  - Strips with `predecessor_cue_id` set render a thin connector line back to the predecessor's right edge — Q-Lab visual style for follow chains.
- Click strip → right-rail drawer with full cue detail + edit form + reaction comments.
- Double-click empty time slot → quick-add cue at that time + lane.
- Drag strip vertically → reassign lane. Horizontally → reschedule. Pinch to zoom time axis (1m / 5m / 15m / 1h tick density). Cmd-drag → duplicate.
- Snap-to-grid: 1 minute snap by default; Shift-drag for free positioning.

### 3.2 Show mode (COMPVSS primary, also available in ATLVS for control room)

**Primary view: Q-Lab cursor.** Single-screen, full-bleed. Three vertical strips:

| Position                              | Content                                                                                                                                                                                                                                              | Behavior                                                                                                                                                                                                                      |
| ------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Top — Recent**                      | Last 3 fired cues, smallest type. Each row: time fired (mono, secondary), lane chip, cue label. Scrolls up as new cues fire.                                                                                                                         | Read-only history strip; click any row → drawer with the cue detail + "Reopen" if appropriate.                                                                                                                                |
| **Middle — Live (xl) + Standby (lg)** | Live cue: huge type (3rem+), lane chip, owner avatar, running clock since `live` transition, GO duration if autofollow set. Underneath: Standby cue (lg type) with countdown `T-MM:SS` until expected fire. Both cards span the full viewport width. | Live card shows `--color-error` accent border. Standby card shows `--color-warning` (amber) accent. Both visible simultaneously — the calling SM needs both in one glance.                                                    |
| **Bottom — Action Bar**               | Single huge **GO** button (full-width, ≥80px tall, color = `--color-error`). To the left: smaller "Standby Next" button. To the right: smaller "Skip" + "Hold" buttons. Above: 2-row queue preview (next 5 pending cues, compact).                   | Space bar → GO (keyboard). Long-press 1s on GO → confirm-double-tap to fire (prevents fat-finger on touch). "Hold" pauses autofollow timers across the show without changing state. "Skip" advances current cue to `skipped`. |

Top chrome:

- Show clock (running): time since show start (mono, large). Toggleable to time-of-day.
- Filter chip strip: lane (multi — usually pinned to your department), event (when multi-show day).
- Mode toggle ([Plan | Show]) hidden during show mode by default — operator dropdown reveals it (prevents accidental mode flip mid-show).

Allowed `DataViewKind` set (plan mode): `["board" /* strip board, custom — see §10 */, "table", "timeline", "print"]`. Default: strip board. Show mode allows: `["show"/* the Q-Lab cursor */, "inbox", "history"]`. Default: show cursor.

The `board` kind is **overloaded** for ROS — the registry's "board" usually means Kanban; here it means strip board. To keep the registry honest, this spec adds a new `DataViewKind = "strip"` to `src/components/views/DataViewKind.ts` and uses it for the ROS plan mode primary. Kanban is reserved for state-grouped boards (Surfaces #1/#2/#4).

Filter chips (plan mode):

- Lane (multi) — quick department filter.
- Owner (typeahead) — your-cues filter.
- Event (multi, only when project has >1 event) — multi-show day.
- State (multi) — usually used post-show to see what got skipped.
- Time window (preset chips: Today · This Show · Next 60 min · Hour-by-hour).

Saved views:

- "Lighting Only" — lane = lights.
- "Talent Standbys" — lane = talent; sort by scheduled_at.
- "Today's Show" — event = current_show (resolved at server time); collapsed lane mode hidden.
- "Skipped Last Show" — state = skipped; event window = previous show.

## 9. Mobile / narrow viewport behavior

**Show mode is mobile-native.** SM on the floor with a phone/tablet is the canonical user. Plan mode degrades on mobile but is not the use case.

### 9.1 ATLVS console at narrow viewport

- Plan mode: strip board collapses to a single-lane mode (operator picks the lane in a chip), horizontal scroll preserved. Editing degrades to a bottom sheet form.
- Show mode: identical chrome to /m/ros — same component, same keys.

### 9.2 COMPVSS /m/ros (mobile-primary show mode)

- Bottom navigation tab "ROS" (already wired in `MobileTabBar`).
- Full-bleed Q-Lab cursor. Live card top-half; Standby card middle; GO button + Skip/Hold bottom.
- Pinch in/out on the live card → reveal the show clock + autofollow next-fire timer.
- Swipe up from the GO button → reveal recent-fired history (5 rows).
- Pull-to-refresh → manual reconciliation if `RealtimeRefresh` lags.
- `prefers-reduced-motion` → no live pulse; replace with a 1-Hz hard-edge color flip on the live border (same signal density, no smoothness).

### 9.3 COMPVSS /m/ros/[showId]

Same Q-Lab cursor but scoped to a single event. The bottom queue preview shows ONLY cues with `event_id = [showId]`. Spec ensures the existing route exists; if it currently just dumps a flat list, the rewrite uses the show-mode component with a filter.

### 9.4 GVTEWAY portal — talent set-time view

- `/p/[slug]/<persona>/cues` — vertical card list. Each card: scheduled time (big), label, owner, countdown if within 60 minutes. State pill only when `live` (then the card pulses red, full attention signal). No GO affordance — read-only.
- Push: 60-min, 30-min, 15-min, 5-min, and standby pre-fire notifications. `PushKind = "cue_countdown"` with `lead_minutes` in payload.
- Talent quick action on a card: "I'm ready" — flips a `cues.talent_ready_at` JSON field (additive, opt-in column — see §12). Owner sees green checkmark on plan-mode strip.

## 11. Acceptance — the business decision the surface exists for

Phase 2 lands tests asserting:

1. **Stage manager calls GO in 1 tap from /m/ros during a live show.**
   - Open /m/ros (mobile, today's show resolved automatically) → show-mode cursor shows the live cue + standby cue → tap GO → standby cue becomes live → push fans out to all viewers of the show → next pending cue becomes new standby. Total: 1 tap from app open to fire. (Fast-confirm mode for trained SMs.)

2. **Autofollow chain fires in sequence.** Author cue A with `autofollow=true, autofollow_after_seconds=30, predecessor_cue_id=null`; author cue B with `autofollow=true, autofollow_after_seconds=0, predecessor_cue_id=A`. Fire A → A goes live → 30s later A done + B live automatically. No manual intervention.

3. **Talent gets standby push 5 minutes before set.** Cue scoped to a talent_id, scheduled_at = T. At T-5min, `cue_countdown` push fires with `lead_minutes=5`; talent's /p/[slug]/cues card pulses.

4. **Strip board reorder persists.** Drag cue from 17:00 to 17:30 on lights lane; release; refresh → cue is at 17:30. `audit_log` records the drag (existing audit trigger handles it).

5. **Multi-author plan-mode conflict resolved gracefully.** User A opens the cue edit drawer; user B edits the same cue and saves; user A's drawer banner: "Reloaded — {user B} updated this cue." Form fields highlight yellow with old/new diff.

6. **Disconnect + reconcile preserves intent.** Fire GO on /m/ros with network off → toast "Fired locally"; reconnect → `cue_transitions` row written with `fired_at` from the local click time + a `synced_at` from server reconcile time. If another SM fired the same cue offline at a different time, the earlier `fired_at` wins; loser sees a conflict-resolution toast.

7. **PDF export renders.** GET `/api/v1/projects/[id]/ros/export.pdf?lane=lights&day=2026-05-24` returns a one-page PDF with the lighting cue sheet for that day.

8. **LiveBeacon updates ≤2s after fire.** Fire a cue in one tab; another tab (different user, viewing /console — not ROS) shows the beacon pulsing within 2s.

## 12. Resolutions — 2026-05-24

Future-proof resolutions baked in. Where a downstream question surfaces in implementation it is captured here, not deferred.

1. **Single mode or split surfaces?** **Single surface, top-level mode toggle.** Reasoning: a separate URL for show mode (e.g. `/console/production/ros/show`) would force duplicate data plumbing and a moded URL is harder to deep-link than a `?mode=show` query string. The toggle is hidden during active show mode to prevent accidental mid-show clicks. Future flexibility: if show mode evolves into a kiosk-style full-screen app, host it on the same route with a `?kiosk=1` modifier rather than a new route.
2. **Transition log — append-only `cue_transitions`?** **YES, ship it.** This is the single biggest gap in the current schema and the foundation for post-show retro, autofollow telemetry, and reconciliation logic. Schema in §10. Append-only. RLS org-scoped. Triggered on every `cues.status` change via `tg_cue_state_transition` so existing UI paths automatically gain the log without code edits.
3. **Autofollow columns on `cues` vs separate `cue_followups` table?** **Columns on `cues`.** Reasoning: a one-to-one relationship (each cue has zero or one predecessor) belongs on the row. A separate table would create N+1 reads on every page render. Forward-flexibility: if conditional follow chains (cue A → either B or C depending on a runtime flag) ever land, they go into a `cue_followups` join table at that point; the additive columns here don't preclude that.
4. **Designated SM per event?** **`events.stage_manager_id` column.** Forward-compatible with multiple SMs per event (the column type stays uuid; future-extension to `events.stage_manager_ids uuid[]` is additive). Reasoning: assigning SM globally per-org is wrong (SMs change per event); assigning per-cue is too granular (one SM calls dozens of cues). Per-event is the right grain.
5. **Live cue accent color in Bermuda Triangle theme?** **Universal `--color-error` red, NOT a shell accent.** Reasoning: the "live" signal must be unambiguous across shells. ATLVS pink + COMPVSS yellow + GVTEWAY cyan are brand colors; red is a runtime-state color. Universal-red live signal sits on top of brand accents without conflict.
6. **Show mode kiosk lock — should leaving show mode require unlock?** **Soft lock only — mode toggle hidden in a settings menu during show.** Reasoning: a hard kiosk lock (password, fullscreen) is overkill for the 95% case and a frustration when the SM needs to escape to fix something. Soft lock = the toggle exists but is one extra step. Future: an explicit "Lock to Show" toggle in event settings for productions that want kiosk mode (TV networks, etc.).
7. **GO button — single press or confirm?** **Per-event setting `fire_confirm_mode`.** Default = `confirm` (long-press 1s OR double-tap Space within 500ms). Senior SMs can switch to `fast`. Reasoning: shop-floor stage managers we've talked to split 50/50 on this preference; making it per-event lets each show tune to its SM. Future-flexibility: per-SM preference column (`users.fire_confirm_mode`) can override per-event default if/when needed.
8. **Realtime disconnect — what happens to a queued offline GO that conflicts with another SM's GO?** **Earlier `fired_at` wins; loser shown conflict toast with the option to re-fire if appropriate.** Reasoning: time-ordering by intent (when SM pressed the button) is the only honest reconciliation; "server arrival time" rewards latency. Future-flexibility: when production-grade conflict resolution is needed (multiple SMs in the same booth), introduce a primary-SM lock at the event level.
9. **Strip board zoom — what's the default time-tick density?** **15 minutes for a typical show day.** Operator can pinch / button-toggle 1m / 5m / 15m / 1h. Default per-user preference saved to `view_configs.metadata.ros_zoom_default`. Reasoning: 15min is the most common rhythm for show cues that aren't autofollow-chained.
10. **Talent "I'm ready" model — column on cues or join table?** **`cues.talent_ready_at timestamptz NULL`** scalar column. Single-talent cues are the dominant case (95%+). Multi-talent cues (e.g. group set call) can use a JSON `talent_ready_log jsonb` future-extension column — additive, won't break the scalar consumers. Reasoning: a separate `cue_talent_ready` join table is overengineered for the common case.
11. **Multi-show day support — `event_id` should be NOT NULL?** **No — keep NULL-able but surface a "Scope" required field in the cue authoring UI.** Reasoning: making `event_id` NOT NULL would break existing org-wide cues (doors-open, dinner-break) that legitimately don't belong to a single event. The surface forces the operator to pick "Event-scoped" or "Project-wide" at authoring time, defaulting to the current event when one is selected — this fills `event_id` reliably without a NOT NULL constraint.
12. **PDF export — per-lane or per-show?** **Both.** Query parameter `lane=…` filters; absence means all-lane sheet. Same endpoint covers both.
13. **`<LiveBeacon>` — does it surface in every shell or just COMPVSS?** **Every shell.** A live signal in the platform top-bar is universally useful — admin seeing it from /console/finance knows the show is live and that they shouldn't push a database migration right now. Forward-flexibility: shells can override the click target (ATLVS goes to control-room ROS, COMPVSS goes to /m/ros, GVTEWAY portal goes to talent set-time view).
