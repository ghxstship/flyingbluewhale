**Shell:** ATLVS admin (pipeline + Rippling drawer) + COMPVSS field (my onboarding) + GVTEWAY portal (counterpart view of their own engagement)
**Route:** /console/people (consolidated engagement directory) + /console/people/[personId] (the Rippling drawer) + /console/people/offer-letters + /console/people/msas + /console/workforce/onboarding (template authoring) + /m/onboarding (assignee checklist) + /p/[slug]/<persona> (counterpart-side engagement view)
**Status:** Drafted · awaiting review — stop signal per brief
**Theme:** Bermuda Triangle only. ATLVS pink on console, GVTEWAY cyan on portal, COMPVSS yellow on /m surfaces. Accents via `--org-primary`.

**Three-layer truth, one surface.** An Engagement = `(Party, Project, Role-class, Channel)` with a `lifecycle_state` that runs through 9 macro-arc values. Inside an engagement live **documents** (offer letter, MSA, signed riders, NDAs) each with their own `document_state` machine, and **onboarding tasks** (template-driven and engagement-specific) each with their own `step_state` machine. The surface is the **consolidator** — the place where these three lifecycles braid into a single per-person view.

## 3. Primary view (per shell)

The surface diverges across shells like Surface #4 (Deliverables). **One spec, three primary views, one engagement record.**

### 3.1 ATLVS admin — Pipeline Board (default) + Rippling drawer (detail)

**Primary list view: Pipeline Board (Kanban grouped by `engagement_state`).** Filter chip strip allows scope by role_class (Performer / Crew / Vendor / Sponsor / Guest / etc.) and channel. Default route: `/console/people` (today a flat directory — repurposed to consolidator).

Lane layout — **9 lanes flatten to 4 super-lanes** (same pattern as Productions/Deliverables):

| Super-lane | Member states                    | Operator question this answers                                                        |
| ---------- | -------------------------------- | ------------------------------------------------------------------------------------- |
| **COURT**  | discovered · interested · vetted | Who are we still evaluating? When does each one need a decision?                      |
| **LOCK**   | committed · enabled · confirmed  | Who's said yes but isn't legally locked? What's the missing document/onboarding step? |
| **RUN**    | active                           | Who's actively engaged on a project right now?                                        |
| **END**    | closed · archived                | Who completed their engagement? (Default-collapsed.)                                  |

Super-lane mode default; `?lanes=flat` toggle exposes the 9-lane flat view for deep-week planning.

Card chrome:

- Top: Party name + avatar + `<RoleClassChip>` (chip in role-class accent — see §5 for the 23-value palette).
- Middle: Project name (small mono) · Channel chip · `<DueDateBadge>` for the next blocking deadline (signature deadline, onboarding deadline, start date).
- Bottom: `<StatusBadge state={uis.lifecycle_state} />` · doc-count chip (e.g. "2/3 docs") · onboarding-progress chip (e.g. "5/8 steps") · "Stuck N days" red chip if state-update is older than `role_class`-specific SLA (configurable per org).
- Left border 3px per Bermuda canon. Tinted by role-class accent rather than state (state is the lane, not the card color).
- Card click → right-rail **Rippling drawer** (see below).

Adjacency rules: forward = adjacent state only; backward = within super-lane only; cross-super-lane backward (e.g. confirmed → discovered) requires the detail page with reason. Same pattern family as Surfaces #1/#2/#4.

**Detail surface — Rippling drawer / page** at `/console/people/[personId]`:

Three vertical sections in a scrollable column, plus a sticky right rail with quick actions:

1. **Identity block** (top): name, photo, contact info, role-class chips, primary project chip, `<PhaseStepper enum="engagement_state" current={…} onAdvance={…} />` showing the 9-state arc with the current state highlighted.
2. **Engagement timeline** (middle): every (Party × Project × channel) row this person has across the org, each rendered as a collapsible card with a mini-stepper, document chips, onboarding progress, transition-log peek. Collapsed by default; the current/active one expanded.
3. **Documents pane**: every doc tied to this person — offer letters, MSAs, signed riders, NDAs, COI. Each row: type · counterpart · state pill · share link · "open" button.
4. **Onboarding pane**: every assignment (template-based + engagement-attached), grouped by source. Inline progress bar + jump-to-step.
5. **Activity timeline** (always-visible right rail or bottom on narrow): unified `LdpStateTimeline` consuming `engagement_state_transitions` ∪ `offer_letter_activity` ∪ `msa_state_transitions` ∪ `onboarding_step_state_transitions` — the whole story by `transitioned_at desc`.

Sticky quick-action rail (right edge): "Advance Engagement" · "Send Offer Letter" · "Send MSA" · "Assign Onboarding Flow" · "Add Note".

### 3.2 GVTEWAY portal — Counterpart engagement view

Counterpart = the Party themselves (a contractor, a vendor contact, an artist's manager). Their question: **"What do I owe and what am I waiting on?"**

Route: `/p/[slug]/<persona>/engagement` mounted on every persona where engagement applies (vendor, crew, artist, sponsor, delegation, hospitality, media). The page renders a per-persona vertical list:

- Top: `<PhaseStepper enum="engagement_state" compact />` showing where they are in the arc (read-only).
- "What I owe" section: documents pending their signature (offer letters in `sent` or `viewed`, MSAs in `sent`) + onboarding steps assigned to them in `pending` or `in_progress`.
- "What we're waiting on" section: documents pending the org's signature (`countersigned` pending), org-side onboarding steps owned by their counterpart.
- "Done" section: completed documents + steps with timestamps.

Affordances per row: tap → submission surface (re-use the per-deliverable submission surface pattern from Surface #4 for documents; mobile-friendly checkbox flow for steps).

### 3.3 COMPVSS field — My Onboarding (assignee checklist)

`/m/onboarding` (existing) is the canonical COMPVSS surface. Extends today's flow-list with a per-assignment detail page (`/m/onboarding/[assignmentId]` — already routed) that becomes the **BambooHR checklist**:

- Top: flow name · progress (`5/8`) · big circular progress ring.
- Body: vertical step list. Each step row: title · `<StateBadge>` · due date · critical-path star · description (expandable) · primary action button (one of: "Read", "Sign", "Upload", "Complete").
- Per-step actions are state-aware:
  - `pending` → "Begin"
  - `in_progress` → "Mark Done" (with required artifact if step is `requires_evidence`)
  - `done` → checkmark, no action
  - `waived` → "Waived by {admin}" muted
  - `blocked` → "Blocked: {reason}" + "Ask manager" button
- Realtime: `RealtimeRefresh` on the assignment row → live updates if a manager waives or unblocks remotely.
- Push: assignment created → `PushKind = "onboarding_assigned"`. Step waived/blocked by manager → `PushKind = "onboarding_step_changed"`. (Already CLAUDE.md plumbed.)

## 9. Mobile / narrow viewport behavior

### 9.1 ATLVS console at narrow viewport

- Board → super-lane mode forced; horizontal snap-scroll between super-lanes.

## 11. Acceptance — the business decision the surface exists for

Phase 2 lands tests asserting:

1. **Admin can advance an engagement from `vetted` to `committed` in ≤2 clicks from any console route.**
   - ⌘K → "advance {party.name}" → Enter → next-state dialog → confirm. `engagement_state_transitions` row written; mirror channel tables' states updated by trigger if applicable.

2. **Rippling drawer renders the full per-person arc.**
   - Open /console/people/[personId] → identity + N engagement cards + M doc rows + K onboarding assignments + activity timeline UNION view all visible in one scroll. <300ms render time for a person with 10 engagements + 30 docs (server-render with parallel queries).

3. **Offer-letter countersign auto-advances the engagement.**
   - Mark `offer_letters.status='countersigned'` → trigger updates `uis_roles.lifecycle_state` to `confirmed`. Pipeline board reflects the move within 30s via realtime.

4. **New hire flow assignment fires push.**
   - Assign `new_hire_flow` to a user → `notifications` row + push to user with `kind='onboarding_assigned'`.

5. **Checklist completion on /m/onboarding cascades.**
   - Tap "Mark Done" on the last critical-path step in a flow → assignment completes → admin's /console/workforce/onboarding flow card shows 100% within 30s → engagement_state_transitions log records the cascade.

6. **Stuck signal fires.**
   - An engagement in `vetted` state with `updated_at` older than the per-role-class SLA (default 7d for production_crew) renders the red "Stuck Nd" chip. Hover → "Last advanced from {from} on {date}."

7. **Bulk send offer letters respects policy.**
   - Select 12 vetted engagements; bulk-send offer letter → 12 offer_letters rows created in `draft` state → admin re-confirms send for each in the dialog → on confirm, all 12 transition to `sent` and engagements advance to `committed`.

8. **Counterpart portal flow.**
   - Portal user (artist) opens /p/[slug]/artist/engagement → sees "What I owe" with their offer letter pending signature → taps "Sign" → completes signature via existing offer-letter public token flow → engagement advances + admin sees the move on the pipeline board within 30s.

## 12. Resolutions — 2026-05-24

Future-proof resolutions baked in.

1. **Engagement state — `uis_roles.lifecycle_state` or invent a new column?** **Use `uis_roles.lifecycle_state`.** It ships, it's the LDP-canonical macro arc, and it sits at the right grain (Party × Project × channel). The LDP audit §5 verdict ("DISTRIBUTED") is **stale** — the unified column exists. Spec drives adoption: every engagement-touching UI reads + writes `uis_roles.lifecycle_state`; per-channel mirror tables (talent_offers etc.) keep their states for channel-specific richness, with triggers keeping the macro arc in sync.
2. **Mirror tables — collapse them into `uis_roles` or keep them?** **Keep them.** Reasoning: a `talent_offers` row carries fee structure, rider attachments, and contract terms that don't belong on a generic `uis_roles` row. The macro arc lives on `uis_roles`; the per-channel detail lives on the mirror table. Triggers keep the macro arc state in sync with the mirror state. Future-flexibility: when a new channel ships (e.g. `sponsor_proposals`), a new mirror table + a new trigger is all that's needed.
3. **`<RoleClassChip>` color palette — 23 colors or fewer?** **6-color palette grouped by class family.** 23 distinct colors is unreadable. Grouping: leadership/executive (pink), talent/performer (pink-secondary), vendor/build/hospitality (amber), crew/production/operations (cyan), sponsor/marketing (green), guest/vip/member (purple), support/staff (grey). Reasoning: color encodes class family, label encodes the specific class. Same approach Frame.io uses for the file-type icon family.
4. **Two onboarding mechanics — consolidate or keep?** **Keep both.** They answer different questions: `new_hire_flows` is org policy (template-based, reusable across hires); `onboarding_steps` is engagement-specific (attached to a specific offer letter, e.g. "this contractor on this project needs to sign these riders"). Consolidating would force the engagement-specific case to inherit a template, which is the wrong shape. Forward-flexibility: a `new_hire_flow.engagement_attached boolean` column could mark flows that ARE engagement-specific if/when the two converge — additive, doesn't break the current split.
5. **Documents — surface them inside the engagement record or keep their own routes?** **Both.** `/console/people/offer-letters` + `/console/people/msas` stay as flat lists for the document-owner workflow ("show me every MSA in `sent` state across the org"). The Rippling drawer surfaces docs scoped to the person ("show me Sarah's documents"). Same data, two lenses — matches the segmented-control pattern from Surface #4 Resolution #1.
6. **Document-state TypeScript type drift — fix in this spec?** **Yes — small but real.** The `offer_letters` schema enum has 11 values (per migration 0019); `src/lib/offer-letters/types.ts` lists 7. Fix in the surface composition list above. Future-flexibility: a build-time check `npm run gen:types` regenerates these from the schema; the drift is a process gap not a design choice.
7. **Activity timeline cost — one subscription or four?** **One subscription per page** over the new `v_engagement_activity` UNION view. Same pattern as Surface #4 Resolution #8. Future-flexibility: when new activity sources ship (e.g. comments-on-engagements), they add an arm to the UNION view, not a new subscription to the client.
8. **`engagement_advance_policy` — separate table or extend the unified `phase_advance_policy`?** **Extend the unified table.** Surface #2 Resolution #7 introduced `phase_advance_policy` with a `phase_kind` discriminator covering both project and production. Add `phase_kind='engagement'`. Three lifecycles, one policy table. Each new lifecycle that needs a policy = one more discriminator value, zero new schema infrastructure.
9. **Bulk send dialog — confirm-per-row or one-shot?** **One-shot send with a confirmable preview.** Operator selects 12 engagements + template → preview dialog lists all 12 recipients + bound variables (rate, dates) computed from each engagement's `project_id` + role config → operator confirms once → all 12 send. Per-row confirm is friction without value. Future-flexibility: per-row override (custom rate for engagement 7 of 12) is a row-level edit in the preview, not a separate flow.
10. **Counterpart portal — one route per persona or one shared route?** **One route per persona** (`/p/[slug]/artist/engagement`, `/p/[slug]/vendor/engagement`, etc.). Reasoning: the persona-scoped portal navigation already maps persona → route prefix; adding a separate "engagement" sub-route per persona keeps the URL structure consistent. The page implementations are thin wrappers over a shared `<CounterpartEngagementView>` component, so duplication is minimal.
11. **"Stuck" SLA — global default or per-org-per-role-class?** **Per-org-per-role-class.** Different orgs vet talent on different cadences; a global default would feel wrong for half of them. New `engagement_sla_policy` table seeded with reasonable defaults (7d for production_crew, 30d for vetted talent, 60d for vendor). Override per row.
12. **Comment thread — same as Surface #4's `<CommentThread targetTable={...} target_id={...} />`?** **Yes.** That primitive was generalized in Surface #4 Resolution #7 precisely for this case. Targets: engagements (target_table='uis_roles'), offer letters (target_table='offer_letters'), MSAs (target_table='msas'), onboarding steps (target_table='new_hire_step_assignments'). Storage adapter routes to the appropriate per-table comments table OR a future generalized `record_comments` polymorphic table. Decision deferred to whichever record ships its comment thread first — for now the primitive accepts the target reference and the storage adapter handles routing.
